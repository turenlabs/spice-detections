# 2026-06-01 Miasma: The Spreading Blight (Red Hat npm)

A credential-stealing, self-propagating worm pushed into the `@redhat-cloud-services` npm namespace
on 2026-06-01. At least 32 package releases (~80,000 weekly downloads combined) were backdoored.
The campaign — "Miasma: The Spreading Blight" — is a new variant of the Mini Shai-Hulud family
derived from the tooling TeamPCP open-sourced; the modifications are largely cosmetic, swapping the
Dune theme for Greek mythology ("spartan"). Attribution to TeamPCP is TTP overlap, not definitive —
the public tooling makes a copycat plausible.

## Vectors

- **Initial access**: a Red Hat employee GitHub account / CI was compromised. Packages were published
  via **GitHub Actions OIDC** (`id-token: write`) from a workflow named `release` triggered on push to
  any branch — i.e. a CI/CD compromise, not a stolen npm token. Malicious commits landed on transient
  `oidc-<8hex>` branches (e.g. `oidc-61fff775`, `oidc-af10000d`).
- **Install-time execution**: each poisoned `package.json` gained `"preinstall": "node index.js"`,
  running a ~4.2 MB obfuscated payload before any application code. The workflow form invokes
  `bun run _index.js` with an `OIDC_PACKAGES` env var.
- **Obfuscation**: multi-stage decryption — numeric char arrays, a ROT-style transform, `eval()`, and
  AES-128-GCM blobs — dropping a transient Bun-based payload (`/tmp/p*.js`).
- **Collection**: broad sweep of CI tokens (`GITHUB_TOKEN`, `ACTIONS_RUNTIME_TOKEN`), AWS/GCP/Azure
  credentials, Vault tokens, Kubernetes service-account tokens/kubeconfig, npm/PyPI publish tokens,
  SSH/GPG keys, Docker creds, and `.env` files. New in this variant: dedicated **GCP and Azure
  cloud-identity collectors** (queried with user-agent
  `google-api-nodejs-client/7.0.0 gl-node/20.11.0 gccl/7.0.0`).
- **Exfiltration**: disguised to `api.anthropic.com/v1/api` (not a real Anthropic route — pure
  camouflage), plus a GitHub **dead-drop**: public repos created under victim accounts with the
  description `Miasma: The Spreading Blight`, holding stolen creds as JSON result files.
- **Persistence**: `kitty-monitor.service` (Linux) / `com.user.kitty-monitor.plist` (macOS); hooks
  injected into AI dev tools (Claude, Codex, Gemini, Copilot, Kiro, opencode) and `.vscode/tasks.json`
  folder-open tasks; payload dropped to `~/.config/index.js`.
- **Dead-man switch**: `gh-token-monitor` watches the stolen GitHub token and can wipe the home
  directory if the token is revoked before persistence is removed — isolate hosts and remove
  persistence *before* rotating tokens.
- **Propagation**: republishes backdoored versions of any package the victim account can access.

## Added coverage (`miasma-2026-06`)

- **Affected packages CSV**: 32 `@redhat-cloud-services/*` packages with exact compromised versions
  (Wiz's list, which includes the re-published patch versions from both waves). This is the primary,
  profile-independent detection — it matches against `package.json`/lockfiles, which are always
  content-scanned.
- **Standalone IOCs**: the `Miasma: The Spreading Blight` repo marker; the `api.anthropic.com/v1/api`
  camouflage endpoint; the GCP-collector user-agent; the `OIDC_PACKAGES` workflow env; and the pinned
  malicious `oven-sh/setup-bun@0c5077e…` action ref.
- **Composite — preinstall loader**: `@redhat-cloud-services/` scope + `"preinstall": "node index.js"`
  in the same `package.json` (minMatches 2). Tight and specific; legitimate Red Hat versions do not
  carry this hook.
- **Composite — OIDC release workflow**: `bun run _index.js` + `OIDC_PACKAGES` + `id-token: write` +
  the two pinned attacker action SHAs (minMatches 3).
- **Composite — payload harvester**: campaign beacon + AES decrypt chain + cloud/CI credential sweep +
  Bun drop + `kitty-monitor`/`gh-token-monitor` persistence + GitHub dead-drop repo write
  (minMatches 3).

## Notes

- **No file hashes** were published (only the ~4.2 MB payload size), so `knownSha*` are empty.
- `suspiciousFilenames` is intentionally empty: the install artifact is `index.js`/`_index.js`, far
  too generic to flag by name. The named persistence files (`kitty-monitor*`, `gh-token-monitor*`) are
  already carried by `mini-shai-hulud-2026-05`; they appear here only as composite signals to avoid
  duplicate findings.
- The JS-payload and `.github/workflows/*.yml` IOCs/composites only fire where those files are
  content-scanned — i.e. the deep and Shai-Hulud scan profiles (and inside package caches). In the
  default profile, Miasma is caught by the affected-version CSV and the `package.json` preinstall
  composite. Making workflow YAML always-scanned would require an engine change (`isAlwaysScanBase`)
  and is a candidate follow-up shared across the Shai-Hulud family.
- No engine change was required for this pack: every rule type used is already expressible as data and
  runs against file classes the engine content-scans.
